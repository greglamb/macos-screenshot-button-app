# ScreenshotButton v1.1 — Review Follow-ups Design

**Date:** 2026-04-14
**Status:** Approved, ready for implementation plan
**Author:** Greg Lamb (with Claude)

## Summary

Close out seven small items deferred from the v1 final code review. No architectural changes; each item is a scoped edit with its own test where the boundary supports one.

## In scope

1. `URLOpening` protocol injected into `Notifier` so the "Open Settings" deep link is test-verifiable.
2. `MenuView` Autolaunch toggle errors surfaced via `Notifier` instead of swallowed.
3. Eager `Notifier.requestAuthorization` on app launch.
4. `Toggle` label changed from "Autolaunch" to "Launch at Login" (macOS System Settings wording).
5. `FileSink` caches its `DateFormatter` instead of allocating per call.
6. `NSScreen.displayID` helper falls back to `CGMainDisplayID()` (not `0`) when `NSScreenNumber` is missing, with an `os_log(.error)` marker so real regressions are visible.
7. `Casks/screenshotbutton.rb` `zap trash:` entry — audit and drop if the app writes no preferences under that path.

## Out of scope

- `OverlayManager` integration tests (remains in `TODO.md`; larger effort).
- Re-entry handling in `OverlayManager.begin` (remains in `TODO.md`).

## Design per item

### 1. `URLOpening` protocol

**Files:** `ScreenshotButton/Services/Notifier.swift`, `ScreenshotButtonTests/Services/NotifierTests.swift` (new), `ScreenshotButtonTests/Fakes/FakeURLOpener.swift` (new).

Add a narrow `@MainActor` protocol next to `Notifier` (kept Notifier-local per YAGNI — no other caller today):

```swift
@MainActor
protocol URLOpening {
    func open(_ url: URL)
}

@MainActor
struct SystemURLOpener: URLOpening {
    func open(_ url: URL) { NSWorkspace.shared.open(url) }
}
```

`Notifier.init` gains:

```swift
init(opener: any URLOpening = SystemURLOpener()) { ... }
```

The `userNotificationCenter(_:didReceive:)` delegate method routes through `opener.open(url)` instead of calling `NSWorkspace.shared.open(url)` directly. Because the delegate method is `nonisolated async` and hops to the main actor via `await MainActor.run`, the `opener` call stays on the main actor.

**Test:** `NotifierTests.openSettingsAction_routesToScreenRecordingURL` constructs a `Notifier` with a `FakeURLOpener` that records received URLs, synthesizes a `UNNotificationResponse` for `openSettingsAction` (the tricky part — `UNNotificationResponse.init` is private, so we test the delegate logic via a helper method `handle(actionIdentifier:)` extracted from the delegate body and called directly). The helper takes the action identifier string and performs the open; the delegate becomes a one-liner that forwards to it. Assert the recorded URL equals the expected `x-apple.systempreferences:com.apple.preference.security?Privacy_ScreenCapture`.

### 2. MenuView Autolaunch errors via Notifier

**Files:** `ScreenshotButton/UI/MenuView.swift`, `ScreenshotButton/ScreenshotButtonApp.swift`, `ScreenshotButtonTests/Fakes/FakeNotifying.swift` (new), `ScreenshotButtonTests/UI/AutolaunchHandlerTests.swift` (new).

Introduce a small `@MainActor protocol Notifying` that `Notifier` already conforms to (extract from existing public methods):

```swift
@MainActor
protocol Notifying {
    func post(title: String, body: String)
    func postPermissionDenied()
}
```

`Notifier` conforms without code changes (it already has these signatures). `MenuView` gains `let notifier: any Notifying`. `ScreenshotButtonApp` passes the existing `Notifier` instance. The toggle's `onChange` handler, on throw, posts:

- title: `"Couldn't update Launch at Login"`
- body: `"Try again in a moment. If the problem persists, open System Settings → General → Login Items."`

…and reverts the toggle (existing behavior).

**Test target:** Extract the error-handling closure into a small free function or helper value so it can be unit-tested without SwiftUI view instantiation:

```swift
@MainActor
struct AutolaunchToggleHandler {
    let launchAtLogin: LaunchAtLogin
    let notifier: any Notifying
    func setEnabled(_ enabled: Bool) -> Bool {
        do { try launchAtLogin.setEnabled(enabled); return enabled }
        catch {
            notifier.post(title: ..., body: ...)
            return launchAtLogin.isEnabled
        }
    }
}
```

`MenuView`'s `onChange` calls `handler.setEnabled(newValue)` and assigns the result back to `launchAtLoginEnabled`. Tests inject a throwing `FakeSMAppServiceAPI`-backed `LaunchAtLogin` and a `FakeNotifying` that records posts; assert post title/body and reverted return value.

### 3. Eager `Notifier.requestAuthorization`

**Files:** `ScreenshotButton/Services/Notifier.swift`, `ScreenshotButton/ScreenshotButtonApp.swift`.

Promote the private `ensureAuthorization` to `func requestAuthorization() async`, keep the `didRequestAuth` idempotency guard so repeated calls are no-ops. Existing `post` / `postPermissionDenied` continue to call it lazily as a backstop.

In `ScreenshotButtonApp`, fold the eager call into the existing `.task(priority: .background)` on the MenuBarExtra label:

```swift
.task(priority: .background) {
    await notifier.requestAuthorization()
    let dir = URL(fileURLWithPath: NSTemporaryDirectory())
        .appendingPathComponent(FileSink.folderName, isDirectory: true)
    TempCleanup.prune(directory: dir, olderThan: 60 * 60 * 24)
}
```

`notifier` must be captured — make it `@State` or lift the declaration so the closure can reach it. Current code stores `notifier` locally in `init`; lift it to a stored property alongside `controller` and `overlays`.

**Test:** No direct test (hits live `UNUserNotificationCenter`). Manual smoke: launch app cold, verify first-launch auth prompt appears before any capture attempt.

### 5. FileSink cached DateFormatter

**Files:** `ScreenshotButton/Services/FileSink.swift`.

Replace per-call allocation with a cached formatter:

```swift
private static let filenameFormatter: DateFormatter = {
    let f = DateFormatter()
    f.dateFormat = "yyyy-MM-dd-HH-mm-ss"
    f.timeZone = .current
    f.locale = Locale(identifier: "en_US_POSIX")
    return f
}()
```

Add `en_US_POSIX` locale to defend against calendar/locale surprises — the existing formatter had neither, which is a latent bug.

`deliver(_:)` uses `Self.filenameFormatter.string(from: nowProvider())`.

**Test:** Existing `FileSinkTests` round-trip assertion (if present) still passes. No new test.

### 6. `displayID` helper fallback

**File:** `ScreenshotButton/UI/OverlayManager.swift`.

```swift
private extension NSScreen {
    var displayID: CGDirectDisplayID {
        if let number = deviceDescription[NSDeviceDescriptionKey("NSScreenNumber")] as? NSNumber {
            return number.uint32Value
        }
        Logger(subsystem: "dev.greglamb.ScreenshotButton", category: "overlay")
            .error("NSScreen missing NSScreenNumber; falling back to CGMainDisplayID()")
        return CGMainDisplayID()
    }
}
```

`CoreGraphics` is already imported; add `import os` at the file top if not already present.

**Test:** Helper is fileprivate and tied to `NSScreen`; not unit-testable without a mock `NSScreen`. Skip.

### 7. Cask `zap trash:` audit

**File:** `Casks/screenshotbutton.rb`.

Grep the codebase for writes to `UserDefaults.standard`, `UserDefaults(suiteName:)`, or any file write under `~/Library/Preferences/`. Current known writes:

- `LaunchAtLogin` uses `SMAppService.mainApp.register()` / `.unregister()` — **system-managed state**, not a preferences plist under the app's domain.

If grep confirms no app-owned plist is written, **delete** the `zap trash:` block:

```ruby
# remove:
zap trash: [
  "~/Library/Preferences/dev.greglamb.ScreenshotButton.plist",
]
```

If a real plist write is found, leave the cask unchanged and add an item to `TODO.md` documenting what state lives there.

## Risk & rollback

- Each item ships as its own commit; reverting any one is safe in isolation.
- Only item 3 (eager auth) changes observable runtime behavior on launch; worst case is a notification permission prompt appearing once on next launch (benign).
- Item 7 is a distribution artifact change that takes effect on the next release build (no impact until a new tag ships).

## Testing strategy

- New Swift Testing `@Suite`s under `ScreenshotButtonTests/Services/NotifierTests.swift` and `ScreenshotButtonTests/UI/AutolaunchHandlerTests.swift`.
- New fakes: `FakeURLOpener` (records opened URLs), `FakeNotifying` (records posts).
- Coverage: maintain ≥75%. New code paths are covered by the new tests.
- Run locally: `./bin/regen && xcodebuildmcp test-sim` (or `test-mac` for the macOS scheme).
